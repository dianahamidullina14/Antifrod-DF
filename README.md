# Antifrod-DF
Необходимо решить задачу классификации для операций, которые по различным причинам, связанных с безопасностью, были не подтверждены клиентами банка. То есть,  предстоит разработать алгоритм реализующий ядро системы антифрода для предотвращения мошеннических операций. 


## 1. Постановка задачи

В работе решается задача бинарной классификации банковских операций:

* **target = 1** — неподтверждённая операция (потенциальный фрод)
* **target = 0** — подтверждённая операция

Особенности задачи:

* сильный дисбаланс классов,
* временная зависимость данных,
* большой объём данных (сотни миллионов операций),
* необходимость моделирования поведения клиента во времени.

Данные разделены на периоды:

* **Pre-train** — история без разметки (для извлечения признаков),
* **Train** — размеченные данные,
* **Test** — данные, разбитые по неделям (time-based split).

---

## 2. Общий подход

Решение построено как классический pipeline анализа транзакционных данных с учётом временной структуры:

1. Загрузка и анализ данных
2. Предобработка и очистка
3. Формирование признаков
4. Обучение модели классификации
5. Предсказание на тесте

Ключевая идея — использовать **поведенческие признаки клиентов** и агрегаты по истории операций.

---

## 3. Данные

Используются несколько источников:

* `train_labels.parquet` — целевая переменная
* `train_part_*.parquet` — история операций
* `test.parquet` — тестовые данные
* `sample_submit.csv` — формат ответа

### Важно

Датасеты **не включены в репозиторий**, так как:

* имеют большой объём,
* хранятся отдельно (Kaggle / внешние источники),
* загружаются напрямую в ноутбуке.

---

## 4. Анализ данных (EDA)

В ноутбуке проводится первичный анализ:

* проверка пропусков (`isna()`),
* статистики (`describe()`),
* количество уникальных клиентов,
* распределение целевой переменной:

```python
labels['target'].value_counts()
```

Выявлено:

* сильный дисбаланс классов,
* значительная вариативность операций.

## 5. Предобработка данных

### 5.1 Работа с пропусками

* анализируются пропущенные значения,
* принимается решение об их обработке или игнорировании.

### 5.2 Кодирование категориальных признаков

Используется:

```python
LabelEncoder

Причина:

* преобразование категориальных признаков в числовые,
* совместимость с ML-моделями.

---

## 6. Формирование признаков

Основной акцент сделан на:

### Поведенческих признаках клиента:

* количество операций,
* частота операций,
* типы операций,
* каналы (channel_indicator),
* MCC-коды.

### Агрегатах:

* статистики по суммам (`operation_amt`),
* распределение по времени,
* количество операций за период.

### Временных признаках:

* использование `event_dttm`,
* анализ поведения во времени.

Это позволяет модели учитывать динамику действий клиента.

---

## 7. Особенности задачи

### 7.1 Дисбаланс классов

Фрод-операции встречаются значительно реже, что требует:

* аккуратного обучения,
* возможного использования весов классов.

---

### 7.2 Временная зависимость

Важно:

* не допускать утечек данных из будущего,
* учитывать, что модель работает в реальном времени.

---

### 7.3 Большой объём данных

Решение учитывает:

* работу с parquet,
* поэтапную загрузку данных,
* ограничение объёма в памяти.

---

## 8. Обучение модели

Используется классический ML-подход:

* подготовленные признаки подаются в модель классификации,
* обучение проводится на размеченных данных (Train),
* далее выполняется предсказание на Test.

(Конкретная модель может варьироваться — в ноутбуке предусмотрена гибкость).

---

Результат формируется в формате:

```

---

## 10. Причины выбора подхода

Выбранный подход обусловлен особенностями задачи:

1. **Табличные данные**
   → эффективнее использовать классические ML-модели, чем LLM

2. **Временная структура**
   → важны агрегаты и история клиента

3. **Большой объём данных**
   → требуется простая и масштабируемая модель

4. **Интерпретируемость**
   → агрегатные признаки проще анализировать

---

## 11. Ограничения

* качество сильно зависит от feature engineering,
* возможны утечки при неправильной работе со временем,
* дисбаланс классов усложняет обучение.

---

## 12. Вывод

Решение представляет собой классический pipeline для антифрод-задачи:

* анализ транзакционных данных,
* построение поведенческих признаков,
* обучение модели классификации.

Основной вклад — корректная работа с временными данными и признаками клиента, что критично для задач обнаружения мошенничества.
